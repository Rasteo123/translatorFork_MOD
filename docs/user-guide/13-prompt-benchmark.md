# Бенчмарк промптов и моделей

Бенчмарк нужен, чтобы сравнивать промпты и модели на одних и тех же фрагментах текста. Он не заменяет вычитку, но быстро показывает, где модель теряет теги, плейсхолдеры, глоссарий, оставляет CJK-недоперевод или даёт слишком короткий/длинный ответ.

## Когда использовать

Используйте бенчмарк, если вы хотите:

- выбрать между несколькими моделями;
- проверить новый промпт перед переводом всей книги;
- понять, помогает ли глоссарий;
- сравнить обычный и короткий промпт;
- проверить локальную модель перед длинным прогоном;
- собрать проблемные фрагменты и гонять их после изменений в настройках.

## Как открыть интерфейс

1. Запустите приложение.
2. В стартовом меню выберите `Бенчмарк промптов`.
3. Нажмите `Новый` или выберите готовый benchmark JSON в поле `JSON`.
4. Заполните вкладки `Cases`, `Prompts` и `Models`.
5. На вкладке `Запуск` отметьте чекбоксами, какие cases/prompts/models прогонять.
6. Для первой проверки оставьте включённым `Только собрать промпты, без API-запросов`.
7. Нажмите `Запустить`.

Пример готового конфига лежит здесь:

```text
benchmarks/prompt_benchmark.example.json
```

## Быстрый безопасный прогон

Начинайте с режима `prompt-only`. В этом режиме программа только собирает промпты и пишет отчёты, но не обращается к API.

Это полезно, чтобы проверить:

- правильный ли JSON выбран;
- все ли фрагменты попали в промпт;
- подставился ли глоссарий;
- не получился ли промпт слишком большим;
- нет ли ошибки в шаблоне.

После `prompt-only` откройте папку отчёта и посмотрите файлы в `prompts`.

## Поля интерфейса

`JSON` — путь к benchmark-конфигу.

`Новый` — создать новый конфиг из шаблона.

`Загрузить` — прочитать выбранный JSON и заполнить вкладки.

`Сохранить` и `Сохранить как...` — записать текущий состав вкладок в benchmark JSON.

`Название` — имя benchmark-прогона. Оно попадёт в отчёт и имя папки.

`Вывод` — папка, куда будут записаны отчёты. Если оставить пустой, будет создана новая папка внутри `benchmark_results`.

Вкладка `Запуск` — выбор матрицы прогона. В списках `Cases`, `Prompts` и `Models` нужно отметить чекбоксами нужные элементы. Кнопки `Все` и `Нет` быстро включают или выключают список.

Вкладка `Cases` — добавление тестовых фрагментов, исходного текста, эталона, глоссария и проверок.

Вкладка `Prompts` — выбор встроенного промпта или ввод своего шаблона.

Вкладка `Models` — выбор провайдера и модели из списка, ввод `base_url`, `api_key_env` и дополнительных параметров.

Вкладка `JSON` — предпросмотр итогового конфига, который будет сохранён и передан runner.

`Только собрать промпты` — не делать API-запросы.

`Сохранять собранные промпты` — записывать итоговый текст каждого промпта в папку `prompts`.

`Лимит запусков` — ограничение на число комбинаций. Удобно для проверки одного-двух запусков перед полной матрицей.

## Как устроен benchmark JSON

Конфиг состоит из трёх основных блоков:

- `cases` — тестовые фрагменты;
- `prompts` — варианты промптов;
- `models` — модели и провайдеры.

Минимальный пример:

```json
{
  "name": "my-benchmark",
  "prompts": [
    {
      "id": "default",
      "builtin": "default",
      "mode": "project"
    }
  ],
  "models": [
    {
      "id": "gemini-flash",
      "provider": "gemini",
      "model": "Gemini 2.5 Flash",
      "api_key_env": "GEMINI_API_KEY"
    }
  ],
  "cases": [
    {
      "id": "chapter-fragment-1",
      "source_html": "<p>青云宗的弟子低声说：“师尊来了。”</p><!-- MEDIA_0 -->",
      "reference": "<p>Ученик Секты Лазурного Облака тихо сказал: «Наставник пришёл».</p><!-- MEDIA_0 -->",
      "glossary": [
        {
          "original": "青云宗",
          "rus": "Секта Лазурного Облака"
        }
      ],
      "checks": {
        "required": ["Секта Лазурного Облака"],
        "forbidden": ["青云宗"],
        "placeholders": ["<!-- MEDIA_0 -->"],
        "min_similarity": 0.25
      }
    }
  ]
}
```

## Cases

Один `case` — это один тестовый фрагмент.

В интерфейсе case добавляется кнопкой `Добавить` на вкладке `Cases`. Заполните `ID`, вставьте исходный фрагмент в поле `Исходный фрагмент` или выберите файл в `Исходник файл`. Можно выбрать TXT, DOCX, Markdown, HTML/XHTML или PDF; программа извлечёт текст/HTML и вставит его в case. Если есть эталонный перевод, вставьте его в `Эталонный перевод` или выберите `Эталон файл`.

Глоссарий редактируется таблицей `Оригинал / Перевод / Заметка`. Проверки ниже задаются обычными полями: обязательные строки, запрещённые строки, плейсхолдеры, similarity и флаги HTML/CJK/JSON.

Частые поля:

- `id` — короткое имя фрагмента;
- `source_html` — исходный HTML/текст прямо в JSON;
- `source_path` — путь к файлу с исходником, если не хотите держать текст в JSON; поддерживаются TXT, DOCX, Markdown, HTML/XHTML и PDF;
- `reference` — пример хорошего результата;
- `reference_path` — путь к файлу с эталоном;
- `glossary` — термины для этого фрагмента;
- `checks` — проверки качества.

Лучшие cases — это не случайные куски, а проблемные места:

- диалог с кавычками;
- фраза с важным термином;
- HTML со ссылкой;
- фрагмент с картинкой или плейсхолдером;
- CJK-текст, где модель часто оставляет недоперевод;
- место, где прошлый промпт ломал структуру.

## Checks

`required` — строки, которые должны быть в результате.

`forbidden` — строки, которых не должно быть в результате.

`placeholders` — плейсхолдеры, которые нельзя потерять.

`preserve_html_tags` — проверять количество HTML-тегов. По умолчанию включено.

`allow_cjk` — разрешить CJK-символы в результате. По умолчанию выключено.

`min_similarity` — минимальное сходство с `reference`. Это мягкая эвристика, не абсолютная оценка литературного качества.

`min_length_ratio` и `max_length_ratio` — допустимая длина результата относительно исходника.

`expect_json` — требовать валидный JSON-ответ.

## Prompts

Вариант промпта может использовать встроенный шаблон:

В интерфейсе prompt добавляется на вкладке `Prompts`. Для обычного прогона выберите `Встроенный default` и режим `project`. Для собственного короткого шаблона выберите `Свой шаблон` и режим `raw`, затем вставьте текст в поле `Текст промпта`.

```json
{
  "id": "project-default",
  "builtin": "default",
  "mode": "project"
}
```

Или собственный короткий шаблон:

```json
{
  "id": "compact",
  "mode": "raw",
  "template": "Translate into literary Russian. Preserve HTML and placeholders. Return only HTML.\n\nGlossary:\n{glossary}\n\nSource:\n{text}"
}
```

Режимы:

- `project` — прогон через текущий `PromptBuilder`, близко к реальному переводчику;
- `raw` — простая подстановка `{text}`, `{glossary}`, `{format_examples}`, `{previous_chapter_reference}`.

Для сравнения реальных настроек проекта чаще используйте `project`. Для экспериментов с короткими промптами удобен `raw`.

## Models

Модель указывается через провайдера из `config/api_providers.json`.

В интерфейсе model добавляется на вкладке `Models`. Выберите `Провайдер`, затем `Модель` из выпадающего списка. Для локальной OpenAI-compatible модели укажите `Base URL`. Для CLI-прогонов можно заполнить `API key env`, а в обычном интерфейсном запуске бенчмарк также умеет брать ключи, уже сохранённые в приложении.

Пример Gemini:

```json
{
  "id": "gemini-flash",
  "provider": "gemini",
  "model": "Gemini 2.5 Flash",
  "api_key_env": "GEMINI_API_KEY",
  "use_stream": true
}
```

Пример локальной OpenAI-compatible модели:

```json
{
  "id": "local-test",
  "provider": "local",
  "model_id": "local-model",
  "base_url": "http://127.0.0.1:11434/v1/chat/completions",
  "use_stream": false
}
```

В интерфейсе live-запуск может использовать ключи, сохранённые в приложении. В CLI удобнее указывать `api_key_env` и задавать ключ через переменную окружения.

## Отчёты

После запуска создаются:

- `results.json` — полный машинный отчёт;
- `results.csv` — таблица для Excel/LibreOffice;
- `summary.md` — краткий рейтинг и список проблем;
- `prompts/*.txt` — собранные промпты, если включено сохранение;
- `outputs/*.output.txt` — ответы моделей при live-запуске.

В интерфейсе есть кнопки:

- `Открыть summary.md`;
- `Открыть results.json`;
- `Открыть папку`.

## Как читать результат

`Avg score` — средний эвристический балл по паре prompt/model.

`OK` — сколько запусков завершилось ответом модели.

`Errors` — сколько запусков упало из-за API, сети, ключей или ошибок конфига.

`Avg latency` — среднее время ответа.

`Avg input` — примерный размер входного промпта в токенах.

Score не означает «литературно идеально». Он означает, что результат прошёл формальные проверки: не потерял структуру, не оставил запрещённые строки, сохранил глоссарий и плейсхолдеры.

## Рекомендуемый рабочий процесс

1. Нажмите `Новый` или загрузите пример `benchmarks/prompt_benchmark.example.json`.
2. На вкладке `Cases` соберите 5-10 коротких фрагментов из реальных проблем книги.
3. Добавьте в каждый case только самые важные проверки.
4. На вкладке `Prompts` добавьте 1-3 варианта промпта.
5. На вкладке `Models` добавьте модели, которые хотите сравнить.
6. На вкладке `Запуск` отметьте нужную матрицу чекбоксами.
7. Сначала запустите `prompt-only`.
8. Откройте `prompts` и убедитесь, что промпты выглядят правильно.
9. Запустите одну дешёвую или локальную модель с `Лимит запусков = 1`.
10. Если всё работает, снимите лимит и прогоните полную матрицу.
11. Смотрите не только на `Avg score`, но и на `Issues`.
12. Лучшие 1-2 варианта проверьте вручную на целой главе.

## CLI

Собрать промпты без API:

```powershell
python -m gemini_translator.scripts.prompt_benchmark benchmarks\prompt_benchmark.example.json --prompt-only
```

Показать состав конфига:

```powershell
python -m gemini_translator.scripts.prompt_benchmark benchmarks\prompt_benchmark.example.json --list
```

Запустить только одну модель:

```powershell
$env:GEMINI_API_KEY = "..."
python -m gemini_translator.scripts.prompt_benchmark benchmarks\prompt_benchmark.example.json --models gemini-flash-env
```

Запустить только выбранные cases и prompts:

```powershell
python -m gemini_translator.scripts.prompt_benchmark benchmarks\prompt_benchmark.example.json --cases zh-dialogue-glossary --prompts project-default --models gemini-flash-env
```

Задать папку отчёта:

```powershell
python -m gemini_translator.scripts.prompt_benchmark benchmarks\prompt_benchmark.example.json --output-dir benchmark_results\my_run
```

## Частые ошибки

Если `prompt-only` работает, а live-запуск падает, сначала проверьте ключи, модель и сеть.

Если score низкий из-за `missing required terms`, проверьте, есть ли нужный термин в `glossary` и не слишком ли строго написан `required`.

Если score низкий из-за `html tag count changed`, модель меняет HTML-структуру. Усильте инструкцию про сохранение тегов или используйте другой промпт.

Если много `CJK residue chars`, модель оставляет недоперевод. Добавьте `forbidden`, проверьте глоссарий и попробуйте другую модель.

Если `reference_similarity` низкий, это не всегда ошибка. Для литературного перевода возможны разные хорошие формулировки. Используйте similarity как сигнал, а не как единственный критерий.

## Практическое правило

Хороший бенчмарк должен быть маленьким, но злым: несколько коротких фрагментов, каждый из которых проверяет конкретную проблему. Большой случайный набор глав хуже помогает выбирать промпт, потому что шумит и дорого стоит.
