### Что такое `"reset_policy"` и как оно влияет на работу?

Секция `"reset_policy"` в `api_providers.json` — это инструкция для `SettingsManager`, которая объясняет, когда именно нужно сбрасывать "красную карточку" для ключей данного провайдера.

**Как это работает:**

1.  **"Красная карточка":** Когда воркер сталкивается с ошибкой исчерпания суточной квоты (например, `RateLimitExceededError`), `SettingsManager` помечает этот ключ как исчерпанный (`exhausted_level: 2`) и записывает точное время, когда это произошло (`exhausted_at`).
2.  **Проверка при запуске:** Каждый раз, когда вы запускаете новую сессию или открываете виджет управления ключами, `SettingsManager` проверяет все "красные" ключи.
3.  **Применение политики:** Для каждого "красного" ключа он смотрит на `reset_policy` его провайдера и вычисляет, прошло ли уже время сброса.
4.  **"Отбеливание":** Если время сброса прошло, `SettingsManager` автоматически снимает с ключа "красную карточку" (устанавливает `exhausted_at = null`, `exhausted_level = 0`), и ключ снова становится "зеленым" и доступным для работы.

**Таким образом, эта политика позволяет ключам, у которых лимит сбрасывается раз в сутки, автоматически возвращаться в строй без вашего вмешательства.**

### Типы политик и их настройка

Существует два основных типа (`type`) политики сброса:

#### 1. Тип `"daily"` (Ежедневный сброс в определенное время)

Этот тип используется для API (как у Google Gemini), которые сбрасывают лимиты в одно и то же время каждый день в определенном часовом поясе.

**Пример:**
```json
"reset_policy": {
    "type": "daily",
    "timezone": "America/Los_Angeles",
    "reset_hour": 0,
    "reset_minute": 1
}
```

**Описание полей:**

*   `"type": "daily"`: Указывает, что сброс происходит ежедневно.
*   `"timezone"`: **(Критически важно)** Часовой пояс, в котором работает API. Лимиты Google сбрасываются по тихоокеанскому времени (PST/PDT), поэтому здесь указано `America/Los_Angeles`.
*   `"reset_hour"`: Час, в который происходит сброс (в 24-часовом формате). `0` означает полночь.
*   `"reset_minute"`: Минута, в которую происходит сброс. `1` означает, что сброс происходит в 00:01. (Использовать `1` вместо `0` — хорошая практика, чтобы дать серверам API секунду на обновление).

**Как это работает:** Программа берет текущее время, переводит его в часовой пояс `America/Los_Angeles` и смотрит, когда в этом часовом поясе была последняя полночь (точнее, 00:01). Если ключ был помечен как "красный" *до* этого момента, он будет "отбелен".

#### 2. Тип `"rolling"` (Скользящее окно)

Этот тип используется для API (как у OpenRouter или многих других), у которых нет фиксированного времени сброса. Вместо этого лимит действует в течение определенного периода с момента *первого* запроса (например, "1000 запросов в любые 24 часа").

**Пример:**
```json
"reset_policy": {
    "type": "rolling",
    "duration_hours": 24
}
```
**Описание полей:**

*   `"type": "rolling"`: Указывает, что используется скользящее окно.
*   `"duration_hours"`: Продолжительность окна в часах. `24` означает, что ключ будет считаться "красным" ровно 24 часа с того момента, как он получил этот статус.

**Как это работает:** Программа просто берет время, когда ключ получил "красную карточку" (`exhausted_at`), прибавляет к нему `duration_hours` и сравнивает с текущим временем. Если текущее время больше, ключ "отбеливается".

### Как находить и вводить часовые пояса (`timezone`)?

Для поля `"timezone"` используются стандартные имена из **базы данных IANA (tz database)**. Это мировой стандарт для часовых поясов, который учитывает летнее/зимнее время и все исторические изменения.

**Как найти правильное имя:**

1.  **Поиск по городу:** Самый надежный способ — использовать формат `Континент/Крупный_Город`. Например:
    *   `Europe/Moscow` (Москва)
    *   `Europe/London` (Лондон)
    *   `America/New_York` (Нью-Йорк, восточное побережье США, EST/EDT)
    *   `America/Los_Angeles` (Лос-Анджелес, тихоокеанское побережье США, PST/PDT)
    *   `Asia/Tokyo` (Токио)
    *   `UTC` (Всемирное координированное время, без смещений)

2.  **Использование онлайн-списков:** Вы можете найти полный список в Википедии по запросу "List of tz database time zones".

**Правила ввода:**

*   **Точное совпадение:** Имя должно быть введено в точности, с учетом регистра. `america/los_angeles` может не сработать, правильно — `America/Los_Angeles`.
*   **Никаких аббревиатур:** Избегайте использования аббревиатур вроде `PST`, `EST`, `MSK`, так как они могут быть неоднозначными. Всегда используйте полный формат `Континент/Город`.

**Важно:** Для корректной работы политики `"daily"` на компьютере, где запускается программа, должна быть установлена библиотека `pytz` (она есть в вашем `requirements.txt`). Если ее нет, система автоматически откатится к более простому 24-часовому скользящему окну.
